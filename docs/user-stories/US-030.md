# US-030 — SPA wait strategies

## Story
As an **automation tester**
I want `wait_for` to support more than "element visible"
So that SPA flakiness drops.

## Acceptance criteria

### AC-1: strategy parameter
- **Given** the codebase
- **When** I inspect the `wait_for` Action schema
- **Then** it has optional
  `strategy: 'visible'|'network-idle'|'mutation-stable'|'route-change'`
  defaulting to `'visible'`.

### AC-2: network-idle waits
- **Given** a page that fires 3 async XHRs over 2s
- **When** a step calls `wait_for({}, strategy: 'network-idle')`
- **Then** the step succeeds only after the network is idle
  for the configured idle window.

### AC-3: mutation-stable waits
- **Given** a list that re-renders 5 times in quick succession
- **When** a step waits with `mutation-stable`
- **Then** the step returns only after the configured no-mutation
  window passes.

### AC-4: route-change waits
- **Given** a button that triggers `history.pushState` without a real
  navigation
- **When** a step uses `route-change`
- **Then** the step returns when the next pushState fires.

### AC-5: Backwards compat
- **Given** existing scenarios calling `wait_for(target)` without
  strategy
- **When** they run
- **Then** behaviour is identical to PS-001.
