# US-019 — Auth recipe execution + storageState reuse

## Story
As an **automation tester testing a protected site**
I want to provide an auth recipe and have the framework log in once
So that every subsequent crawl / analyse / run sees authenticated content.

## Acceptance criteria

### AC-1: Successful login captures state
- **Given** a valid recipe and credentials
- **When** I run `ai-test run --url https://x/private --auth-recipe inputs/auth/x.yaml --mode explore`
- **Then** a `storage-state.json` is written under the run evidence
  dir with mode `0600`, and the first scenario sees authenticated content.

### AC-2: Wrong password fails fast
- **Given** an incorrect `${SITE_PASSWORD}` env var
- **When** the recipe runs
- **Then** the CLI exits with code `3` and stderr names the recipe path;
  no partial scenarios run.

### AC-3: Session expiry re-login (once)
- **Given** the session expires mid-run
- **When** the next page redirects to login
- **Then** the recipe re-runs once and execution continues; a second
  expiry exits with `AuthExpiredError`.

### AC-4: Credentials never logged
- **Given** any run
- **When** I `grep -r "${SITE_PASSWORD value}" reports/`
- **Then** there are zero matches in JSON, HTML, or ai-trace files.

### AC-5: Recipe references env vars
- **Given** a recipe with `password: ${SITE_PASSWORD}`
- **When** loaded
- **Then** the literal `${SITE_PASSWORD}` is substituted at load time
  from env; a missing env var fails with a clear message.
