# US-026 — axe-core accessibility check

## Story
As a **tester**
I want every scenario to produce an accessibility audit
So that I see WCAG 2.1 AA violations alongside functional results.

## Acceptance criteria

### AC-1: Violations captured
- **Given** any scenario that ends on a page with known WCAG issues
- **When** the run completes
- **Then** `ScenarioResult.accessibilityViolations[]` lists each axe
  violation with `id`, `impact`, `wcagLevel`, `elements[]`,
  `helpUrl`.

### AC-2: WCAG level derived
- **Given** a violation with axe impact `serious`
- **When** mapped
- **Then** the result has `wcagLevel: "AA"` (per documented mapping).

### AC-3: Severity gate
- **Given** `report.a11y.failOn: ["serious","critical"]`
- **When** a scenario hits a `serious` violation
- **Then** the ValidationResult adds a FAIL check named
  `accessibility` and the scenario is marked failed.

### AC-4: Default = warn
- **Given** no `failOn` configured
- **When** any violation occurs
- **Then** check status is `warn`; scenario can still pass.

### AC-5: HTML report panel
- **Given** the run report
- **When** I expand a scenario
- **Then** violations are listed with element selector + remediation
  link.
