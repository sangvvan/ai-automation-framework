# US-018 — Auto-detect login form

## Story
As an **automation tester**
I want the framework to look at a login URL and infer the username +
password fields by itself
So that I do not have to write a recipe for vanilla forms.

## Acceptance criteria

### AC-1: Conventional form detected
- **Given** a page with one text-or-email input + one password input +
  one submit button
- **When** I run `ai-test auth detect --url https://x/login`
- **Then** the CLI prints the inferred field locators (semantic) and
  writes a draft recipe to `inputs/auth/{host}.draft.yaml`.

### AC-2: Ambiguous form aborts gracefully
- **Given** a page with three text inputs and no password input
- **When** detection runs
- **Then** the CLI exits non-zero with
  `Ambiguous login form — provide a recipe manually`.

### AC-3: Field heuristics
- **Given** an input with `name="email"` or `aria-label` containing
  "Email" / "Username" / "Account"
- **When** detection runs
- **Then** the username locator targets that input by role + accessible
  name (not CSS).

### AC-4: Sensitive field flagged
- **Given** the password field
- **When** detection produces the draft recipe
- **Then** the recipe uses `${SITE_PASSWORD}` placeholder, never the
  raw value, even if a value happens to be on the DOM.
