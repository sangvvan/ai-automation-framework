# US-024 — Design-technique field on every scenario

## Story
As a **Test Lead**
I want each scenario to declare which ISTQB design technique produced it
So that I can report coverage by technique.

## Acceptance criteria

### AC-1: Schema field exists
- **Given** the codebase
- **When** I import `ExecutableScenario`
- **Then** it has a non-optional `designTechnique` field of type
  `equivalence-partition | boundary-value | decision-table |
  state-transition | use-case | pairwise | error-guessing |
  exploratory-charter`.

### AC-2: AI generation populates it
- **Given** a run in explore mode
- **When** generation completes
- **Then** every AI-generated scenario has a non-null `designTechnique`.

### AC-3: Tester-supplied scenarios default
- **Given** a YAML test case without `design_technique` key
- **When** parsed
- **Then** the scenario gets `designTechnique: "error-guessing"`
  (ISTQB-CTFL default for tester-experience-driven cases).

### AC-4: Tester can override
- **Given** YAML with `design_technique: state-transition`
- **When** parsed
- **Then** the scenario reflects the supplied technique.

### AC-5: Existing schema backwards-compat
- **Given** a YAML test case from PS-001
- **When** parsed by the new code
- **Then** parsing succeeds (no breaking change).
