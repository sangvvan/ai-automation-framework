# US-038 — Faker-backed test data

## Story
As an **automation tester**
I want realistic synthetic data for AI-generated scenarios
So that tests resemble production usage and locale-specific bugs surface.

## Acceptance criteria

### AC-1: Faker integration
- **Given** an AI-generated scenario with an `email` field
- **When** data is filled
- **Then** the value comes from faker's `internet.email`, not
  `test-${field}`.

### AC-2: Deterministic per run
- **Given** the same runId
- **When** the run is repeated
- **Then** the same fake values are produced (faker seeded from runId).

### AC-3: Locale-aware
- **Given** `runner.locale: vi`
- **When** values are generated
- **Then** addresses and phone numbers follow Vietnamese conventions
  where faker supports it.

### AC-4: Anti-harvest
- **Given** the PageAnalysis contains a visible value
  (e.g. user name "Sang V.")
- **When** values are generated
- **Then** the generator never picks that exact value (deterministic
  collision check).

### AC-5: Sensitive fields masked in report
- **Given** the report
- **When** I view a step that filled a password
- **Then** the masked value is shown, not the generated one.
