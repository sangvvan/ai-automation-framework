# REQ-002 — Test Case Input Mode

## Purpose
Allow a tester to supply test cases in YAML or Markdown. The framework
parses, validates, and converts them into the framework's internal
**ExecutableScenario** representation so the Execution Agent can run them.

## Scope
- Parse YAML test-case files matching the schema documented in PS §6.2.
- Parse Markdown test-case files following a documented heading convention
  (`# TC_*`, `## Steps`, `## Expected result`).
- Validate every parsed case with Zod; surface field-level errors.
- Map natural-language steps (e.g. "Click Login", "Enter valid username")
  to keyword actions (`click`, `fill`, `select`, `verify_text`, …) using
  PageAnalysis context.
- Persist the parsed `ExecutableScenario[]` for traceability.

## Out of scope
- Generating new test cases (covered by REQ-003).
- Recording test cases from a manual run (future).

## Acceptance criteria
- AC-002-1: A YAML test case file matching PS §6.2 schema is parsed
  without error and produces ≥1 ExecutableScenario.
- AC-002-2: A YAML file missing `page_url` or `steps` is rejected with
  a precise validation message (path + reason).
- AC-002-3: A Markdown test case using the documented headings is parsed
  to the same internal shape as the equivalent YAML.
- AC-002-4: Step mapping resolves "Click Login" to a `click` keyword
  against an element whose accessible name matches "Login" on the page.
- AC-002-5: If a step cannot be mapped to a real element, the framework
  records an unresolved-step warning and **does not** silently skip.

## Constraints
- YAML parsing via `yaml` package; never `eval`.
- Reject any test case whose `page_url` host is not in the configured
  allow-list (security guard for CLI users).

## Inputs
- File path(s) to `.yaml`/`.yml` or `.md` test-case file
- PageAnalysis for the target URL (from REQ-001) for step mapping

## Outputs
- `ExecutableScenario[]` validated via Zod
- Validation report (errors + warnings) saved alongside the run
