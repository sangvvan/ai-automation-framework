# REQ-015 — CI/CD & Tooling Interoperability

## Purpose
Make the framework's output consumable by standard CI tooling
(Jenkins, GitLab CI, Azure Pipelines, GitHub Actions) and surface
results directly on pull requests.

## Scope
- **JUnit XML reporter**:
  - New module `app/lib/reporter/junit.ts`.
  - Emits `reports/junit/{run-id}.xml` conforming to the Jenkins
    JUnit schema (`testsuites` → `testsuite` → `testcase` → `failure`
    / `error` / `skipped`).
  - One `<testsuite>` per `TestSuite` from REQ-011; nested `<testcase>`
    per scenario, with system-out for evidence URLs.
- **PR comment integration**:
  - GitHub adapter `app/lib/reporter/github-pr.ts` reads `GITHUB_TOKEN`
    + `GITHUB_REPOSITORY` + `GITHUB_PR_NUMBER` env vars (CI-injected).
  - Posts (or updates) a single comment per PR with: totals, top 5
    failures with short reasons, links to the HTML report artefact and
    JUnit XML.
  - Idempotent: subsequent runs update the same comment (find by
    marker `<!-- ai-test:summary -->`).
- **Exit codes** standardised: 0 (all passed), 1 (assertion failures),
  2 (orchestration error), 3 (auth error / unreachable URL).

## Out of scope
- GitLab Merge Request notes (deferred; same adapter pattern can be
  added when needed).
- Slack / Teams notifications (out of scope; a webhook plugin would
  live outside this REQ).

## Acceptance criteria
- AC-015-1: Every run emits `reports/junit/{run-id}.xml` and the XML
  validates against the canonical JUnit XSD.
- AC-015-2: Suites map 1:1 to `<testsuite>`; scenarios map 1:1 to
  `<testcase>`; the count attributes (`tests`, `failures`, `errors`,
  `skipped`) match the run totals.
- AC-015-3: With `GITHUB_TOKEN` + PR context set, the framework posts
  a comment on the PR containing the run summary; a second run on the
  same PR updates the existing comment rather than creating a new one.
- AC-015-4: With no GitHub env present, the framework skips the PR
  comment silently and exits 0.
- AC-015-5: Failed scenarios yield non-zero CLI exit; CI matrix can
  detect failures via exit code without parsing reports.

## Constraints
- JUnit XML must remain a stable schema (downstream tooling is
  brittle). Add new fields only as `<properties>` children.
- PR comment body capped at 64 KB; truncate failure list when needed
  and link to the HTML report.
- Never log `GITHUB_TOKEN`.

## Inputs / Outputs
- Inputs: `RunSummary` (with TestSuite IDs), CI env vars.
- Outputs: `reports/junit/{run-id}.xml`, posted/updated GitHub PR
  comment, deterministic exit codes.
