# REQ-007 — Human Review & Approval

## Purpose
Provide a reviewer workflow so AI-generated scenarios are not promoted
into official regression suites without a human gate. Reviewers can
inspect, edit, approve, or reject each scenario, and approved scenarios
move from `tests/generated/` into `tests/approved/` (then optionally
into `tests/regression/`).

## Scope
- Web UI in the Remix app:
  - List runs and their generated scenarios.
  - Per-scenario detail: steps, expected result, AI rationale, evidence.
  - Inline edit of step text / expected result.
  - Actions: Approve, Reject (with reason), Edit & Re-run.
  - Bulk approve/reject within a run.
- DB persistence of review state (PostgreSQL).
- Auth: only authenticated reviewers (role = `tester` or `test-lead`) can
  act; unauthenticated users get read-only run summaries.
- Moving an approved scenario into `tests/approved/` writes a
  Playwright-compatible spec stub + a YAML scenario definition.
- An audit trail records who approved/rejected what and when.

## Out of scope
- Multi-project tenancy.
- Approval workflow with multiple sign-off stages (single approver in MVP).

## Acceptance criteria
- AC-007-1: A reviewer with role `tester` can sign in, open a run, and
  see the list of generated scenarios with status `pending_review`.
- AC-007-2: Approving a scenario writes a YAML file under
  `tests/approved/{feature}/{scenario-id}.yaml` and updates the DB row
  to `status = approved` with reviewer id + timestamp.
- AC-007-3: Rejecting a scenario requires a reason ≥10 chars; DB stores
  the reason and the scenario is hidden from default regression runs.
- AC-007-4: A reviewer can edit step text or expected result; the edit
  is versioned (old version retained in the audit trail).
- AC-007-5: An unauthenticated visitor to a run page sees the summary
  but cannot see Approve/Reject controls.
- AC-007-6: Promoting a scenario to `tests/regression/` requires role
  `test-lead`; testers see the control disabled with a tooltip.

## Constraints
- All actions validated server-side with Zod; CSRF protected.
- No bulk approve > 50 items per request.
- Audit log is append-only.

## Inputs
- Generated scenarios (REQ-003) and their evidence (REQ-005/006)
- Reviewer identity (auth)

## Outputs
- DB rows: `scenarios`, `scenario_reviews`, `audit_log`
- Files in `tests/approved/` and (test-lead only) `tests/regression/`
