# PS-002 — Universal-Web + ISTQB Compliance Enhancement

> Builds on PS-001. Captures the gap-analysis (chat 2026-05-17) into a
> formal problem statement for Sprint 4+5+6.

---

## 1. Background

PS-001 delivered the MVP — a single-page AI-driven test automation
framework with CLI, web Review UI, AI generation, and basic execution +
reporting. It successfully runs a tester-supplied YAML test case
end-to-end against a URL (verified by the fixture-app smoke run, May 2026).

Two **strategic gaps** remain before the framework can meet its long-term
vision:

1. The current Page Analyzer reads exactly one URL. Real web applications
   are multi-page, behind authentication, with SPA dynamics. A tester
   cannot just "drop a link and walk away" today.
2. The current reports describe scenarios and results, but no formal
   **Test Plan**, **Test Suite**, or **traceability matrix** in the
   ISTQB sense — and they cover only functional + console-error checks.
   Quality characteristics from ISO/IEC 25010 (performance, usability,
   accessibility, security, compatibility, portability) are not measured.

---

## 2. Problem

### 2.1 Single-page scope
The framework cannot test an entire web application from a single entry
URL. Users must manually enumerate every page, defeating the "give it a
link" value proposition.

### 2.2 No authentication recipe
Protected pages (anything behind login) cannot be analysed or tested
without manual scripting. Real-world apps are mostly protected.

### 2.3 No formal ISTQB artefacts
The framework lacks the documents ISTQB-trained QA leads expect:
Test Plan (scope, approach, entry/exit criteria, risks, schedule),
Test Suite grouping, and a complete Requirements → Test Conditions →
Test Cases → Execution → Defects traceability chain.

### 2.4 Narrow test-design coverage
The AI Test Design Agent enumerates scenario "categories" loosely
(positive / negative / boundary / …) but does not apply the **seven**
ISTQB test design techniques: equivalence partitioning, boundary value
analysis, decision table, state transition, use-case, pairwise, and
error guessing. There is no exploratory charter mode.

### 2.5 Functional-only test types
Only functional behaviour and basic console errors are checked. There
is no accessibility audit (axe-core sits unused in `package.json`),
no performance metrics (Web Vitals / Lighthouse), no security header
checks, no compatibility matrix (Chromium only), no i18n support.

### 2.6 Limited stability
SPA dynamic content uses only `domcontentloaded`. There are no
self-healing locators, no screenshot baseline diffing, no retry policy
ladder, no advanced interactions (file upload, drag-drop, iframe,
shadow-DOM).

### 2.7 CI/CD integration gaps
No JUnit XML output for legacy CI systems (Jenkins / GitLab / Azure
Pipelines). No PR comment integration.

### 2.8 Cost runaway risk
AI generation has a scenario cap but no token budget; a misconfigured
provider could rack up large bills silently.

---

## 3. Goal

Transform the framework so that:

- A tester provides **only** a starting URL (plus optional credentials)
  and receives a complete Test Plan + Test Suite + executed Test Cases
  + traceability matrix covering the **entire reachable application**.
- All artefacts and reports conform to **ISTQB** vocabulary and
  structure (Test Plan, Test Suite, Test Case, Test Run, Test Item,
  Defect, Traceability).
- Test types cover ISO/IEC 25010 quality characteristics — functional,
  performance, usability, accessibility, compatibility, security, and
  (where data exists) reliability and portability.
- Execution is stable across SPA, multi-browser, multi-locale.
- Reports are consumable by both humans (HTML) and legacy CI (JUnit).

---

## 4. Vision (post-enhancement)

```text
Input URL (+ optional credentials)
        ↓
Crawler: discover SiteMap (bounded, polite, robots-aware)
        ↓
Per page: PageAnalysis (with auth state propagated)
        ↓
TestPlan generator: scope, approach, risks, traceability frame
        ↓
TestDesign agent: scenarios per ISTQB technique × test type × risk
        ↓
TestSuite assembler: group, prioritise, deduplicate
        ↓
Multi-browser runner: chromium + firefox + webkit
        ↓
Validators: functional + axe + Web Vitals + security headers + screenshot diff
        ↓
Reports: HTML + JSON + JUnit + PR comment + DB persistence
        ↓
Defects, coverage maps, regression diffs, audit trail
```

## 5. Target Users (delta vs PS-001)

Same four personas (Automation Tester, Manual Tester, Test Lead,
Developer) — but now:

- **Test Lead** gets the Test Plan + Suite-level dashboard with risk and
  coverage roll-up, and can map AC ↔ Test Conditions ↔ Test Cases.
- **Automation Tester** spends time reviewing crawler scope and approving
  generated suites, not enumerating URLs.
- **Developer** sees Web Vitals and a11y violations in the same report
  as functional failures.

## 6. Scope (in scope for this enhancement)

### 6.1 Crawler & SiteMap
Bounded BFS crawl from an entry URL, robots.txt-aware, dedupe by route
pattern, output `SiteMap` + per-page `PageAnalysis`.

### 6.2 Authenticated test flows
Auto-detect login form (or accept a recipe), execute login, persist
storageState, reuse across crawl + scenarios.

### 6.3 Test Plan artefact
Zod-typed `TestPlan` document: scope, levels, types, entry/exit
criteria, risks, schedule, resources, traceability matrix. Generated
from analysis + config + requirements, persisted to `reports/test-plans/`.

### 6.4 Test Suite layer
DB-backed `TestSuite` grouping scenarios with setup/teardown hooks and
a regression tag.

### 6.5 ISTQB-aligned design techniques
Expand `ScenarioType` enum with the seven techniques; update the Test
Design prompt to drive technique-aware generation.

### 6.6 Non-functional test types
- Accessibility: wire `@axe-core/playwright`, attach violations to
  results.
- Performance: inject `web-vitals` lib, capture LCP / CLS / INP / TTFB,
  optional Lighthouse hook.
- Security: response-header validator (CSP, HSTS, X-Frame-Options,
  cookie flags).
- Compatibility: multi-browser matrix (chromium, firefox, webkit).
- i18n: locale-driven re-runs (Accept-Language + `<html lang>`).

### 6.7 Stability
- SPA wait strategies (network-idle, mutation-stable, route-change).
- Self-healing locator with audit trail.
- Screenshot baseline diffing with pixel threshold.
- Extended assertions (visibility, attributes, child count, negation).

### 6.8 Advanced element interactions
Keywords: `upload_file`, `drag_drop`, `switch_to_frame`, shadow-DOM
piercing in resolver.

### 6.9 Test data generation
Faker integration with deterministic seed (per runId), locale-aware,
PII-masked in reports.

### 6.10 CI/CD interop
JUnit XML reporter; GitHub PR comment summariser.

### 6.11 Cost guardrails
Per-run token budget for AI generation; trace logged with token counts;
hard halt at budget exhaustion.

### 6.12 Defect persistence
Promote validator `suggestedDefect` outputs into a `defects` table with
a Review UI drawer; optional external issue-tracker linking left as a
post-MVP-2 follow-up.

### 6.13 Coverage reporting
Element coverage heatmap (tested vs untested UI elements per page);
AC traceability map (acceptance criteria → scenario IDs).

---

## 7. Out of Scope (deferred to a future enhancement)

- SSO / OAuth / SAML — external IdP integration complexity.
- True load / stress / soak performance testing (capacity, not metrics).
- Penetration testing or DAST scanners (basic header checks only).
- Real-device / cloud-grid execution (BrowserStack / Sauce Labs).
- Mobile-native (iOS / Android) testing.
- Automatic Jira / GitHub issue creation from defects (manual linking only).
- Visual baseline-management UI (CLI re-baseline only).

---

## 8. Success Criteria

The enhancement is successful when:

1. A tester runs `ai-test crawl --url https://example.com --auth-recipe
   inputs/auth/example.yaml` and receives a SiteMap covering reachable
   pages within the configured budget.
2. A tester runs `ai-test run --site-map <id>` and receives a Test Plan,
   a set of Test Suites (one per page or feature), executed scenarios
   per ISTQB technique, and a report with functional + a11y + perf +
   security results.
3. Reports include a JUnit XML alongside HTML+JSON so any CI can
   consume them.
4. Multi-browser runs produce a compatibility matrix in the report.
5. The traceability matrix shows Requirement → Test Plan → Test Suite →
   Test Case → Test Run → Defect for every scenario.
6. The Review UI exposes the Test Plan, Suite, and Defect views in
   addition to existing scenario approval.
7. Token usage is bounded per run; the trace shows actual costs.

---

## 9. Constraints

- Must remain backwards-compatible with the PS-001 CLI surface
  (`ai-test run --url ... --mode testcase` continues to work unchanged).
- Web crawler must be polite by default (robots.txt-aware, configurable
  concurrency, per-host rate limit).
- Crawl must not exfiltrate sensitive page content to external AI
  providers unless explicitly authorised.
- All new artefacts use existing storage roots (`reports/`,
  `tests/approved/`, DB) — no new top-level directories beyond
  `reports/test-plans/`, `reports/baselines/`, `reports/sitemaps/`.
- Multi-browser support must not break Chromium-only CI (browser list
  is config-driven).
- No new top-level frameworks (Remix + Playwright + Vitest stack stays).

## 11. Open Questions (to be resolved during `/ps` and `/ba`)

- Crawler scope: stay same-origin only by default, or allow opt-in
  subdomain following?
- Test Plan: which ISTQB-recognised template do we mirror (IEEE 829,
  ISTQB-CTFL syllabus, or a slimmed framework-specific shape)?
- Defect external linking: ship a Jira adapter in this enhancement, or
  defer entirely?
- Multi-browser parallelism: per scenario, per suite, or per run?
- Storage of screenshot baselines: in-repo (`reports/baselines/`) or
  in S3 with a path indirection?
- Token budget: hard halt or soft warn + continue?
