# REQ-005 — Result Validation

## Purpose
For each `ScenarioResult` produced by REQ-004, run validation checks
against expected results and emit a final pass/fail decision plus a
machine-readable defect summary when failing.

## Scope
- Validate: page navigation (URL match), UI element visibility, text
  messages, form validation messages, URL changes, API response status
  (where the scenario records one), console errors, and (optional)
  screenshot diff vs baseline.
- Distinguish "soft" assertions (collect all failures) from "hard" ones
  (fail fast).
- Produce a `ValidationResult` per scenario:
  `{ status, checks[], failureReason?, suggestedDefect? }`.

## Out of scope
- Visual baseline management UI (future).
- Auto-creating defects in Jira/GitHub (PS §14.2 explicit out-of-scope).

## Acceptance criteria
- AC-005-1: A scenario whose expected URL/text matches actual is marked
  `passed` with all checks recorded.
- AC-005-2: A scenario whose expected text is missing is marked `failed`
  with `failureReason` naming the expected vs actual text snippet.
- AC-005-3: Console errors observed during execution are surfaced as
  WARN checks even when the scenario passes its functional asserts.
- AC-005-4: Validation never modifies the page under test.
- AC-005-5: A failing validation produces a `suggestedDefect` with
  summary, steps to reproduce, evidence links, and severity hint.

## Constraints
- Validation runs are deterministic given the same `ScenarioResult` input.
- Screenshot diff threshold is configurable; default 0.1% pixel delta.

## Inputs
- `ScenarioResult` (from REQ-004)
- `ExpectedResult` from the originating scenario

## Outputs
- `ValidationResult` per scenario
- Aggregated `RunValidation` summary
