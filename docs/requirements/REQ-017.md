# REQ-017 â€” Defects, Coverage, and Cost Guardrails

## Purpose
Close the loop: persist AI-suggested defects so reviewers can act on
them, measure element + AC coverage to expose gaps, and prevent
runaway AI cost by enforcing a per-run token budget.

## Scope
### Defect persistence
- DB table `defects` with `id`, `run_id`, `scenario_id`, `summary`,
  `steps_to_reproduce`, `evidence_links`, `severity`,
  `status (open|triaged|fixed|wont-fix)`, `external_ref` (nullable),
  `created_at`, `updated_at`.
- Validator inserts a row per `suggestedDefect` it emits.
- Review UI route `/runs/:id/defects` lists defects with status filter
  and a "Copy as issue body" action (clipboard).
- External issue creation (Jira / GitHub) is left for a future REQ;
  `external_ref` is a free-text field reviewers can fill in.

### Element coverage map
- During execution, the runner records each `PageElement.id` touched
  by any keyword (`click`/`fill`/`select`/`wait_for`).
- Per-page coverage is `touched / visibleInteractive` ratio.
- Aggregated per run and rendered in the HTML report as a heatmap
  table (page Ă— element rows).
- Stored under `reports/coverage/{run-id}.json`.

### AC traceability coverage
- `RunSummary.acCoverage`: `{ acId, scenarioIds[] }[]`.
- Generation populates AC linkage where the agent can identify it;
  reviewers can edit linkage in the Scenario detail page.
- Coverage % per REQ surfaced in the HTML report header.

### Cost guardrails (AI token budget)
- New config: `ai.tokenBudgetPerRun` (default 200_000 tokens).
- Provider adapters return `usage: { inputTokens, outputTokens }` from
  every call.
- `chained provider` aggregates per role; on threshold reach, raises
  `BudgetExceededError`. CLI flag `--budget` overrides config.
- Trace `ai-trace.jsonl` records cumulative tokens after every call.

## Out of scope
- Auto-creating tickets in Jira / GitHub.
- Multi-currency cost translation (we report tokens, not $).
- Coverage targets enforced as a gate (visible only, not blocking).

## Acceptance criteria
- AC-017-1: Every failing scenario emits a `defects` row; the Review
  UI route shows it with status `open`.
- AC-017-2: A reviewer can change a defect's status to `triaged` /
  `fixed` / `wont-fix`; audit log records the change.
- AC-017-3: Element coverage JSON exists per run; HTML heatmap shows
  per-page % and lists untested elements above a threshold.
- AC-017-4: AC traceability map is rendered in the HTML report and
  in the Test Plan (REQ-011) traceability matrix.
- AC-017-5: With `ai.tokenBudgetPerRun: 1000`, a large generation run
  fails fast with `BudgetExceededError` and a partial RunSummary is
  saved with the explanation.
- AC-017-6: Provider trace shows input/output tokens per call and a
  running total.

## Constraints
- Defects table is **not** an issue tracker â€” no comments, no
  assignment, no SLA fields.
- Coverage maps may use the page's PageAnalysis as the denominator
  (so coverage is "interactive elements analysed" â€” not full DOM).
- Token usage is best-effort: providers without usage telemetry
  (some local LM Studio configs) skip the count with a warning.

## Inputs / Outputs
- Inputs: ValidationResult (suggestedDefects), runner-tracked element
  IDs, provider responses.
- Outputs: `defects` rows, coverage JSON + heatmap, token-aware
  `ai-trace.jsonl`.
