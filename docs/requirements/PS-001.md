# PS-001 — AI-Powered System Test Automation Framework for Web Applications

> Source: problem statement provided during onboarding.
> Status: Ready for Phase 0 (`/ps`) — extraction into REQs.

---

## 1. Background

In modern web application development, system testing plays a critical role in
ensuring that the application works correctly from an end-user perspective.
However, traditional system testing requires significant manual effort from
automation testers, including:

- Understanding each page and feature flow
- Writing detailed test cases manually
- Creating automation scripts for each scenario
- Maintaining test scripts when the UI or business logic changes
- Re-running regression tests across multiple releases

As web applications grow in complexity, the testing effort becomes harder to
scale. Automation testers and QA teams spend too much time preparing and
maintaining test scripts instead of focusing on test strategy, risk coverage,
and quality improvement.

To improve testing efficiency and coverage, we want to build an AI-powered
automation framework that supports system testing for web applications by
allowing testers to provide minimal input (page URLs and optional test cases).
The AI should understand the page, generate or execute test scenarios,
validate results, and provide structured test reports.

---

## 2. Problem

System testing for web applications today depends heavily on manual test
design and manual automation script creation. Automation testers write
scripts page by page, flow by flow, and maintain them whenever the application
changes. This creates the following challenges:

### 2.1 High Manual Effort
Testers must manually analyze each page, design test cases, prepare test
data, write automation scripts, and maintain those scripts over time.

### 2.2 Slow Regression Testing
Regression test suites take time to prepare and update. New features may not
be tested quickly enough, and release cycles can be delayed.

### 2.3 Inconsistent Test Coverage
Different testers design tests differently. Edge cases may be missed; UI
validation, navigation, negative scenarios, and error handling are not
covered consistently.

### 2.4 Dependency on Detailed Test Cases
Traditional automation requires predefined test cases. If they are missing
or incomplete, automation coverage is limited. Exploratory testing is still
mostly manual.

### 2.5 High Maintenance Cost
UI changes break existing automation scripts. Test data, selectors, and
expected results require frequent updates. Teams spend significant effort
fixing scripts instead of improving quality.

There is a need for a standardized, scalable, and maintainable framework
that reduces manual testing effort, improves coverage, and lets automation
testers execute system testing with minimal input.

---

## 3. Goal

Develop an AI Automation Framework for Web Application System Testing.

The framework allows automation testers to input:

- A web application URL
- Page URLs or user journey URLs
- Optional test cases for each page
- Optional test data
- Optional acceptance criteria or business rules

Based on this input the AI should be able to:

- Analyze the web page structure and behavior
- Understand available UI elements (forms, buttons, links, dialogs, tables, …)
- Generate test scenarios automatically when test cases are not provided
- Execute provided or AI-generated test scenarios
- Validate actual behavior against expected behavior
- Capture screenshots, logs, traces, and test evidence
- Generate structured test reports
- Support regression testing and re-execution

---

## 4. Vision

```text
Input URL / Test Case / Requirement
        ↓
AI analyzes page
        ↓
AI generates test scenarios
        ↓
AI executes system tests
        ↓
AI validates results
        ↓
AI generates report and evidence
```

The framework supports two main modes:

### 4.1 Test-Case-Driven Automation
The tester provides test cases. The AI maps test steps to web UI actions,
executes them, validates expected results, and generates evidence.

### 4.2 AI-Driven Exploratory Automation
The tester provides only a page URL or user flow. The AI analyzes the page
and generates positive, negative, boundary, UI, navigation, validation, and
error-handling scenarios.

## 5. Target Users

### 5.1 Automation Testers
- Input page URLs
- Provide test cases when available
- Let AI generate test cases when not available
- Review generated scenarios
- Run automated tests

### 5.2 Manual Testers

- Convert manual test ideas into executable test scenarios

## 6. Core Capabilities

### 6.1 Page Analysis
Analyze a target web page and identify:
- Page title and purpose
- Main UI sections
- Buttons, links, input fields, dropdowns, checkboxes, radio buttons,
  tables, dialogs, and forms
- Required and optional fields
- Validation messages
- Navigation paths and error states
- Dynamic content
- Accessibility-related attributes where possible

### 6.2 Test Case Input Mode
Support tester-provided test cases (YAML / Markdown). Example:

```yaml
page_url: "https://example.com/login"
test_cases:
  - id: TC_LOGIN_001
    title: "Login successfully with valid user"
    priority: P1
    steps:
      - Open login page
      - Enter valid username
      - Enter valid password
      - Click Login
    expected_result: "User is redirected to the dashboard"
```

The AI converts the test case into executable actions and validations.

### 6.3 AI Test Generation Mode
When no test case is provided, the AI generates scenarios from the page
analysis, covering:
- Positive scenarios
- Negative scenarios
- Boundary scenarios
- Required field validations
- Navigation checks
- UI consistency checks
- Error handling checks
- Basic accessibility checks
- Basic security-related checks (e.g. invalid input handling)

### 6.4 Test Execution
Execute tests using **Playwright** (recommended), with the framework
designed so other engines (Selenium, Cypress) could be plugged in.

### 6.5 Result Validation
Validate:
- Page navigation
- UI element visibility
- Text messages
- Form validation
- URL changes
- API response status where applicable
- Console errors
- Screenshot comparison where needed

### 6.6 Reporting
Generate structured reports with:
- Test summary
- Passed / failed / skipped test cases
- Failure reasons
- Screenshots
- Execution trace
- Logs
- AI-generated observations
- Suggested next actions
- Coverage summary by page, feature, or user journey

### 6.7 Human Review and Approval
For important flows the framework allows testers to:
- Review generated test scenarios
- Approve or reject scenarios
- Edit test steps before execution
- Mark scenarios as reusable regression tests

## 7. Agent-Based Workflow

The framework is designed with multiple AI agents:

| Agent | Responsibility | Input | Output |
|-------|----------------|-------|--------|
| Page Analyzer Agent | Analyze the target page | URL, DOM, screenshot | Page structure, important elements, possible user actions, risks |
| Test Design Agent | Generate test scenarios | Page analysis, optional test cases, optional AC | Scenarios, steps, expected results, priority, type |
| Test Execution Agent | Convert scenarios into automation | Scenarios, locators, test data | Execution result, trace, screenshots, logs |
| Validation Agent | Check actual vs expected | Expected, actual UI state, screenshot, logs, URL, response | Pass/fail decision, reason, possible defect summary |
| Report Agent | Produce final reports | Execution + validation result, evidence | HTML report, JSON report, Test Lead summary, defect suggestions |

---

## 8. Testing Standards and Patterns

- **Page Object Model** — each page has a page object / abstraction layer.
- **Data-Driven Testing** — test data is separated from test logic.
- **Keyword-Driven Testing** — reusable keywords such as
  `open_page`, `click_button`, `fill_text`, `select_dropdown`,
  `verify_text`, `verify_url`, `upload_file`.
- **Risk-Based Testing** — generated scenarios prioritized as P1 / P2 / P3.
- **Human-in-the-Loop Control** — AI-generated test cases reviewable
  before being added to official regression.

## 9. Example User Flows

### Flow 1 — Tester Provides Test Cases
1. Tester provides page URL and test cases.
2. AI analyzes the page.
3. AI maps test steps to UI elements.
4. Framework executes tests.
5. Framework validates expected results.
6. Framework generates report and evidence.

### Flow 2 — Tester Provides Only Page URL
1. Tester provides page URL only.
2. AI analyzes the page structure.
3. AI generates possible test scenarios.
4. Tester reviews and approves scenarios.
5. Framework executes approved scenarios.
6. Framework generates report and suggested defects.

## 10. Constraints

- **Tech direction**: TypeScript + Playwright + Playwright Test;
  YAML/Markdown inputs; HTML + JSON + screenshot + trace outputs;
  Node.js CLI; CI/CD via GitHub Actions or Jenkins.
- **Security & data privacy**: avoid sending sensitive data to external
  AI providers; support local LLM / enterprise-approved AI providers;
  mask credentials and personal data in logs and reports.
- **Human-in-the-loop**: AI-generated scenarios must be reviewable before
  being promoted into official regression suites.
- **Maintainability**: modular architecture, Page Object Model,
  separation of test data / test logic / locators / reports.

## 12. Recommended First MVP

```text
Input:
- Web page URL
- Optional test case in YAML or Markdown

AI Actions:
- Open page
- Analyze UI elements
- Generate test scenarios if no test case is provided
- Execute test using Playwright
- Capture screenshot and trace
- Generate report
```

### MVP Features
- CLI command to run a test
- Page analysis using Playwright
- Test scenario generation
- Playwright test execution
- HTML report
- JSON result
- Screenshot evidence
- Manual approval option for AI-generated test cases

Example commands:

```bash
ai-test run \
  --url "https://example.com/login" \
  --test-case "inputs/test-cases/login.md" \
  --mode "testcase"

ai-test run \
  --url "https://example.com/login" \
  --mode "explore"
```

## 13. Non-Functional Requirements

- **Maintainability** — clear component responsibilities; easy to extend / debug.
- **Scalability** — support multiple applications, pages, and teams.
- **Reliability** — minimize flaky results via reliable locator strategies
  and stable wait mechanisms.
- **Security** — protect sensitive test data, credentials, screenshots, logs.
- **Extensibility** — pluggable integration with CI/CD pipelines, test
  management tools, defect tracking tools, local LLM providers, and
  enterprise AI providers.

---

## 14. Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| AI hallucination (incorrect steps / expected results) | Require human review before promoting to regression; ground AI on actual DOM/browser state; retain evidence for verification. |
| Flaky tests due to dynamic UI / unstable locators | Robust locator strategy (role/text-based); careful retry & wait; no hardcoded sleeps. |
| Security & data privacy | Don't send sensitive data to external AI; support local/enterprise LLM; mask credentials and PII in logs/reports. |
| Incorrect coverage assumption | Apply risk-based testing; Test Lead defines mandatory flows; combine AI-generated and human-designed cases. |
| Maintenance complexity | Modular architecture; Page Object Model; clear agent responsibilities. |

---

## 15. Success Criteria

The framework is successful when:

- Automation testers can run system tests by providing only page URLs and
  optional test cases.
- The AI generates meaningful scenarios when test cases are missing.
- Test execution produces reliable evidence (screenshots, traces, logs).
- Test results are understandable for testers, developers, and Test Leads.
- Generated scenarios can be reviewed and promoted into regression suites.
- The framework reduces manual test design and automation effort.
- The framework can be extended to multiple projects and web applications.
