# REQ-012 — ISTQB-Aligned Test Design Techniques

## Purpose
Make the Test Design Agent generate scenarios that explicitly map to
the seven ISTQB-recognised design techniques, so coverage by technique
is reportable.

## Scope
- Extend `ScenarioType` enum to include the seven techniques:
  `equivalence-partition`, `boundary-value`, `decision-table`,
  `state-transition`, `use-case`, `pairwise`, `error-guessing`.
  (Existing `positive` / `negative` / `required-field` / `navigation` /
  `ui` / `error-handling` / `accessibility` / `security` stay as test
  **types** orthogonal to technique.)
- New schema field on `ExecutableScenario`:
  `designTechnique: TechniqueEnum | null`.
- Update Test Design Agent prompt to accept a `requestedTechniques[]`
  list and generate per technique. Default: all seven plus
  exploratory-charter when no AC is supplied.
- Add an `exploratory-charter` mode: time-boxed AI run with a
  natural-language charter goal (e.g. "Find ways to break this form
  with invalid input"). Output: list of observations + suggested
  scenarios.
- Report rolls up coverage per technique (count + pass-rate).

## Out of scope
- Domain-specific test data generation (covered by Sprint 6 faker work).
- AC parsing into Equivalence Partitions automatically (we ask the LLM
  to propose them, but no formal solver).

## Acceptance criteria
- AC-012-1: Generation against a page with at least one input renders
  ≥ 1 scenario per technique among
  `equivalence-partition`, `boundary-value`, `error-guessing`.
- AC-012-2: Every generated scenario carries `designTechnique` set to
  one of the enum values (never null when origin = `ai-generated`).
- AC-012-3: The HTML report includes a "Technique coverage" panel:
  rows per technique with count + pass-rate.
- AC-012-4: An exploratory-charter run records a `charter` string in
  the RunSummary and lists `observations[]` distinct from
  `scenarios[]`.
- AC-012-5: `requestedTechniques` config can restrict generation to a
  subset; agent does not produce other techniques.

## Constraints
- Existing scenario types (positive / negative / …) remain valid;
  they are exposed as `testType` instead of `scenarioType`.
- Backwards compat: old YAML test cases without `designTechnique`
  default to `error-guessing` (the catch-all per ISTQB-CTFL).

## Inputs
- PageAnalysis (REQ-001)
- Optional `requestedTechniques: TechniqueEnum[]`
- Optional charter string for exploratory mode

## Outputs
- `ExecutableScenario[]` with `designTechnique` populated
- `RunSummary.techniqueCoverage[]` aggregate
- `RunSummary.observations[]` for charter runs
