## Purpose
Produce the formal documents ISTQB-trained QA leads expect: a Test
Plan per run, Test Suite grouping for scenarios, and a complete
traceability matrix from Requirement â†’ Test Condition â†’ Test Case â†’
Test Run â†’ Defect.

## Scope
### Test Plan
- Zod-typed `TestPlan` document with sections:
  scope, test items (URLs / features in scope), test levels, test
  types, approach, entry criteria, exit criteria, risks, schedule,
  resources, deliverables, traceability matrix.
- Generated from: SiteMap + REQ files + framework config.
- Persisted to `reports/test-plans/{run-id}.json` and rendered into
  the HTML report as a tab.

### Test Suite
- DB table `test_suites` with `id`, `run_id`, `name`,
  `feature_slug`, `preconditions`, `setup_hook`, `teardown_hook`,
  `regression_tag`.
- Scenarios reference a suite via `suite_id` (nullable for ad-hoc).
- A suite has at least one scenario; orphan scenarios attach to a
  default `Ad-hoc` suite per run.

### Traceability matrix
- Per-run JSON + HTML table linking:
  `REQ â†’ TestCondition â†’ TestCase (scenario) â†’ TestRun â†’ Defect`.
- "Test Condition" = a paragraph extracted from the AC text used by
  the Test Design Agent as prompt grounding.
- Defect = a persisted suggestedDefect (REQ-017).

## Out of scope
- IEEE 829 full template (we adopt a slim, ISTQB-CTFL-aligned shape).
- Multi-project roll-up (per-run only in this version).

## Acceptance criteria
- AC-011-1: Every run with `--mode crawl` emits
  `reports/test-plans/{run-id}.json` validating against the TestPlan
  Zod schema with non-empty `scope`, `entryCriteria`, `exitCriteria`,
  `risks`, `resources`, and `traceabilityMatrix`.
- AC-011-2: The HTML report shows a "Test Plan" tab rendering all
  Plan sections.
- AC-011-3: Every scenario in a multi-suite run has a `suite_id`;
  suite-less scenarios are auto-attached to an `Ad-hoc` suite.
- AC-011-4: The traceability matrix lists, per REQ, at least one
  Test Condition (extracted from the corresponding US's ACs) and
  one Test Case (scenario id).
- AC-011-5: A scenario marked failed appears in the traceability
  matrix's Defect column once REQ-017 persistence runs.

## Constraints
- TestPlan generation is deterministic given the same inputs (no AI
  call required); AI is used only for narrative summary fields.
- Suites must not cross runs (no FK to other runs).
- Traceability matrix is read-only after run completion.

## Inputs
- `SiteMap` (REQ-009)
- REQ + US markdown files in `docs/`
- FrameworkConfig

## Outputs
- `TestPlan` JSON
- `test_suites` rows + `scenarios.suite_id` populated
- Traceability matrix JSON + HTML render
