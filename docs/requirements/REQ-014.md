# REQ-014 â€” Execution Stability & Resilience

## Purpose
Reduce flaky failures and false positives by adding richer wait
strategies for SPA dynamics, a self-healing locator pass when the
primary locator fails, screenshot baseline diffing, and a broader
assertion vocabulary.

## Scope
### SPA wait strategies
- Extend `wait_for(target)` to accept
  `strategy: 'visible'|'network-idle'|'mutation-stable'|'route-change'`
  with sensible defaults.
- `mutation-stable` watches the DOM for N ms (configurable, default
  500ms) with no mutations.
- `route-change` waits for the next browser-side navigation
  (history.pushState etc.).

### Self-healing locators
- When the primary `resolveLocator` fails:
  1. Capture the visible DOM snapshot.
  2. Re-derive candidate locators (fuzzy text match, partial role,
     attribute swap).
  3. Try them in deterministic order; on first success, record a
     `SelfHealEvent` to the run's audit trail and proceed.
  4. If all fail, raise the original failure unchanged.
- Healed scenarios are flagged in the report (`healedSteps[]`) and
  the validator may downgrade to WARN.

### Screenshot diff baselining
- New keyword `verify_screenshot(name, [threshold])`.
- On first run, capture as baseline under
  `reports/baselines/{suite-slug}/{scenario-id}/{name}.png`.
- Subsequent runs use `pixelmatch` to diff against baseline;
  fail when delta exceeds `screenshotDiffThreshold`.
- Provide CLI subcommand `ai-test baselines accept --run-id ...` to
  promote new baselines.

### Extended assertions
- `ExpectedResult` Zod schema gains:
  - `urlNotContains?: string`
  - `textNotContains?: string`
  - `visibleLocators?: Locator[]`
  - `notVisibleLocators?: Locator[]`
  - `attribute?: { target: Locator; name: string; equals?: string;
      contains?: string; }`
  - `childCount?: { target: Locator; min?: number; max?: number }`
- Validator dispatches new check types.

## Out of scope
- Full visual regression UI (only CLI baseline accept).
- Network mocking / stubbing (use Playwright route() in tester
  scenarios only).

## Acceptance criteria
- AC-014-1: A scenario using `wait_for(target, strategy: 'mutation-stable')`
  succeeds on a known-dynamic page where the default `visible` wait
  flakes (verified with a fixture).
- AC-014-2: When the primary locator fails but a fuzzy match exists,
  the scenario passes; the run report lists the heal event with old
  vs new locator.
- AC-014-3: First run of a `verify_screenshot` step produces a baseline
  file; second run with no UI change passes; UI shift > threshold fails.
- AC-014-4: Each new assertion (visibleLocators, attribute, childCount,
  etc.) has an accept + reject Vitest test against a fixture.
- AC-014-5: `ai-test baselines accept --run-id <id>` overwrites the
  matching baselines and logs the change.

## Constraints
- Self-heal is opt-in via `runner.selfHeal: true` (default false to
  preserve deterministic PS-001 behaviour).
- Screenshot baselines are masked: sensitive fields are blurred before
  capture using the analyzer's sensitivity flags.
- No locator may heal across `kind` boundaries (a `role` locator can
  only heal to another `role`).

## Inputs / Outputs
- Inputs: scenario step actions, RunnerConfig flags.
- Outputs: extended `StepResult.healEvent?`, baseline files, richer
  `ValidationCheck[]`.
