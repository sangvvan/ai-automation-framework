# REQ-004 ŌĆö Test Execution Engine

## Purpose
Take a list of `ExecutableScenario` (from REQ-002 or REQ-003) and execute
each step against a real browser using Playwright, capturing screenshots,
trace files, console logs, network logs, and per-step results.

## Scope
- Provide a Keyword Action Library (`open_page`, `click`, `fill`, `select`,
  `verify_text`, `verify_url`, `upload_file`, `wait_for`, ŌĆ”).
- Use semantic locators (role/name/text/testId); never CSS classes / XPath.
- Persist a per-run directory under `reports/evidence/{run-id}/{scenario-id}/`.
- Capture Playwright trace (`trace.zip`), screenshot on failure (always)
  and on success (configurable), console log, network HAR if enabled.
- Honour test-data injection from `inputs/test-data/`.
- Support sequential and parallel execution (configurable worker count).
- Support tagged re-execution (regression subset).

## Out of scope
- Selenium / Cypress backends (Playwright only in MVP).
- Mobile browser emulation.
- Distributed execution across machines.

## Acceptance criteria
- AC-004-1: A scenario whose steps all map to valid actions runs to
  completion and produces a `ScenarioResult` with `status = passed|failed`.
- AC-004-2: On any step failure, the runner halts that scenario, captures
  a failure screenshot + trace, and continues with the next scenario.
- AC-004-3: All keyword actions resolve element locators in this order:
  byRole(name) ŌåÆ byLabel ŌåÆ byText ŌåÆ byTestId. CSS / XPath are rejected.
- AC-004-4: Each ScenarioResult includes per-step status, duration,
  evidence paths, and any captured console errors.
- AC-004-5: Runs are reproducible: the same scenario + same test data on
  the same app version produces identical step results modulo timing.
- AC-004-6: A scenario that exceeds the configured `stepTimeoutMs` fails
  with reason `timeout` rather than hanging.

## Constraints
- Never `page.waitForTimeout` ŌĆö assertion-based waits only.
- No raw `page.*` calls in scenario layer ŌĆö must go through Keyword Actions.
- Headless by default; `--headed` flag for local debugging.

## Inputs
- `ExecutableScenario[]`
- `TestRunConfig` (browser, viewport, baseUrl, workers, timeouts)
- Test data files

## Outputs
- `ScenarioResult[]`
- Evidence directory tree (screenshots, traces, logs, HAR)
